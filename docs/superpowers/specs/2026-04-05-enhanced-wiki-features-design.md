# Enhanced Wiki Features — Design Spec

## Problem

wikigen produces directory-level summaries that help LLMs navigate to the right part of a codebase, but falls short on helping them *work* in it. An LLM that finds the right directory still doesn't know:
- Which files change together for a given task
- How directories depend on each other (blast radius)
- How to look up a file by name without tree traversal
- What the critical execution paths are through the codebase
- Which packages are public API vs internal
- What testing patterns to follow
- Where key symbols are defined
- Which areas are actively changing

Additionally, the HTML output has broken navigation — child directories aren't linked, so humans can't browse up and down the tree.

## Features

### Feature 1: HTML Tree Navigation (bug fix)

**Problem:** Child directories listed in SUMMARY.html aren't clickable. The only upward navigation is breadcrumbs.

**Solution:** Add an explicit nav block to every SUMMARY.html with links to parent and children. This is structural, not dependent on LLM-generated content.

- `writeHTMLPage()` gains a `childDirs []string` parameter
- New `buildChildNav()` function generates `<nav class="children">` with links to `{child}/SUMMARY.html`
- Appended after body content, before "Generated by wikigen" footer
- Breadcrumbs already handle upward nav — no change needed

### Feature 2: Dependency Graph (inlined in wiki.md)

**What:** Text-based adjacency list showing directory-to-directory dependencies.

**How:** No extra LLM call. After all summaries are generated, parse each summary's "## Dependencies" section, extract `Imports:` and `Imported by:` lines, build a map, emit as a section in wiki.md.

**Format:**
```
pkg/auth → pkg/db, pkg/config
pkg/api → pkg/auth, pkg/db, pkg/middleware
cmd/server → pkg/api, pkg/config
```

Annotated with boundary info (Feature 6) when available:
```
pkg/auth (public) → pkg/db (internal), pkg/config (public)
```

**Opt-out:** `--no-deps-graph`

### Feature 3: File Index (inlined in wiki.md)

**What:** Flat table mapping every source file to its directory and one-liner.

**How:** No extra LLM call. Parse each directory's "## Key files" section, extract `- [filename](./filename) — description` entries, aggregate into a markdown table sorted by filename.

**Format:**
```
| File | Directory | Description |
|------|-----------|-------------|
| handler.go | pkg/api | HTTP request handlers |
| store.go | pkg/db | Database persistence layer |
```

**Opt-out:** `--no-file-index`

### Feature 4: Modification Recipes (separate file: recipes.md)

**What:** "To add X, touch files A, B, C" — derived from git co-change history.

**How:** Two steps:
1. Git co-change analysis (no LLM): `git log --name-only`, build co-occurrence matrix, cluster top 10-15 groups
2. One LLM call: send clusters + directory summaries, get named recipes

**Where:** `recipes.md` + `recipes.html` at wiki root. wiki.md gets a one-liner link.

**Fallback:** Skip if not git repo or < 50 commits.

**Opt-out:** `--no-recipes`

### Feature 5: Entry Point Traces (separate file: traces.md)

**What:** 2-3 critical end-to-end paths traced from entry points through the codebase.

**How:** One LLM call. Send all directory summaries, ask LLM to identify main entry points and trace flows naming specific files and functions.

**Where:** `traces.md` + `traces.html` at wiki root. wiki.md gets a one-liner link.

**Opt-out:** `--no-traces`

### Feature 6: Interface/Boundary Annotations (in directory summaries)

**What:** Per directory: public (exported API), internal (implementation), or entry point (binary).

**How:** No extra LLM call. Add a `## Boundary` section to the existing directory summarization prompt.

**Where:** Each SUMMARY.md. Also aggregated into the dependency graph (Feature 2) as annotations.

**Opt-out:** `--no-boundaries`

### Feature 7: Test Location and Pattern Hints (in directory summaries)

**What:** Per directory: where tests live, testing pattern, helpers/fixtures.

**How:** No extra LLM call. Add a `## Testing` section to the existing directory summarization prompt.

**Where:** Each SUMMARY.md only. No wiki.md aggregation.

**Opt-out:** `--no-test-hints`

### Feature 8: Symbol Index (inlined in wiki.md)

**What:** Key exported types/interfaces and their locations.

**How:** No extra LLM call. Parse each directory's "## Key types and interfaces" section, extract symbol names and locations.

**Format:**
```
| Symbol | Kind | Location |
|--------|------|----------|
| PolicySet | type | pkg/policy/types.go |
| Compile() | func | pkg/cedar/compiler.go |
```

**Opt-out:** `--no-symbol-index`

### Feature 9: Churn Overlay (inlined in wiki.md)

**What:** Directories ranked by recent change frequency (last 90 days).

**How:** No LLM call. `git log --since=90d --name-only`, count commits per directory, classify as high/medium/low.

**Fallback:** Skip if not git repo or < 30 days history.

**Format:**
```
| Directory | Commits (90d) | Churn |
|-----------|---------------|-------|
| pkg/api | 47 | high |
| pkg/auth | 23 | medium |
```

**Opt-out:** `--no-churn`

## Cost Summary

| Feature | Extra LLM calls | Extra git analysis | Opt-out flag |
|---------|-----------------|-------------------|--------------|
| HTML tree nav | 0 | No | N/A (bug fix) |
| Dependency graph | 0 | No | `--no-deps-graph` |
| File index | 0 | No | `--no-file-index` |
| Modification recipes | 1 | Yes | `--no-recipes` |
| Entry point traces | 1 | No | `--no-traces` |
| Boundary annotations | 0 | No | `--no-boundaries` |
| Test hints | 0 | No | `--no-test-hints` |
| Symbol index | 0 | No | `--no-symbol-index` |
| Churn overlay | 0 | Yes | `--no-churn` |

**Total new LLM calls:** 2 (recipes + traces). Everything else piggybacks on existing calls or is pure data aggregation.

## Architecture

### Data Flow

```
1. Existing flow (unchanged):
   collectDirs → computeHashes → detectChanges → propagateDirty
   → for each dirty dir: buildBundle → summarize(LLM) → write SUMMARY.md/html

2. New post-processing (after all summaries exist):
   a. Parse all summaries → extract dependencies, key files, symbols
   b. Git analysis → co-change clusters, churn stats
   c. LLM call → recipes (from clusters + summaries)
   d. LLM call → traces (from all summaries)
   e. Aggregate into wiki.md sections + separate files

3. HTML generation (modified):
   writeHTMLPage now receives childDirs for nav generation
```

### wiki.md Structure (final)

```markdown
# Codebase Wiki

{rootOverview}          ← existing

## Contents             ← existing
...

## Dependency graph     ← NEW (Feature 2)
...

## File index           ← NEW (Feature 3)
...

## Symbol index         ← NEW (Feature 8)
...

## Recent activity      ← NEW (Feature 9)
...

## Modification recipes ← NEW (Feature 4, link only)
...

## Entry point traces   ← NEW (Feature 5, link only)
...

---
## {top-level dir}      ← existing per-dir sections
...
```

### Modified LLM Prompt Sections

The directory summarization system prompt gains two new optional sections:

```
## Boundary
State whether this directory is: **public** (exported API, meant to be imported
by other packages), **internal** (implementation detail), or **entry point**
(binary/command, not imported). One sentence on the contract.

## Testing
Where tests live (colocated, separate dir). Testing pattern (table-driven,
fixtures, mocks, integration vs unit). Name test helpers or shared fixtures.
Skip if no tests.
```

### New Files Generated

| File | Content |
|------|---------|
| `recipes.md` | Modification recipes |
| `recipes.html` | HTML version |
| `traces.md` | Entry point traces |
| `traces.html` | HTML version |

### Incremental Behavior

- Features 2, 3, 6, 7, 8 are regenerated whenever any directory summary changes (they parse summaries)
- Feature 4 (recipes) is regenerated when git history changes (new commits since last run)
- Feature 5 (traces) is regenerated when any directory summary changes
- Feature 9 (churn) is regenerated on every run (cheap git command)
- The manifest gains fields for tracking recipe/trace hashes to support incremental

## Opt-out Flag Design

All flags follow the `--no-{feature}` convention. When a flag is set:
- For prompt additions (boundaries, test hints): the section is omitted from the LLM prompt
- For post-processing (deps graph, file index, symbol index, churn): the section is skipped in wiki.md
- For separate files (recipes, traces): the file is not generated, the LLM call is skipped
